# Contributing

Thanks for contributing to SDK Parity.

## Public Scope

Public repositories should contain reusable open-source code, synthetic fixtures, and user-facing documentation only. Do not include private infrastructure details, customer material, private planning, credentials, or internal strategy.

## Pull Requests

1. Open an issue or draft PR for substantial changes.
2. Keep changes focused and include tests for behavior changes.
3. Run the repository's documented checks before requesting review.
4. Update public documentation when user-facing behavior changes.

## Development

Each repository owns its local setup instructions. Use the commands in that repository's README and package scripts.

## Reporting Security Issues

Do not open public issues for vulnerabilities. Follow the process in `SECURITY.md`.
